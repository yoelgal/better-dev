# Per-vuln-class red-now criteria

One row per work-item type. Each check is written **red now** - the diff fails it until the code satisfies
it - which is the same shape a done-criterion has, so `/plan-grill`'s threat-surface pass can lift a row
straight into the contract and the fallback pass can walk it against the diff. A row applies only when the
diff has that surface; skip the rest. Depth here, the lean gate in `SKILL.md`.

| Work-item type | Red-now checks (green when each holds) |
|---|---|
| **User input / injection** | every external field is validated against a schema at the boundary and rejected when it fails, not coerced into something acceptable; queries are parameterized, never assembled by string concatenation (SQL, NoSQL, LDAP, OS command, path); output is encoded for the exact sink it lands in (HTML body, attribute, URL, shell argument). |
| **Auth / session** | authorization is checked on every protected route, not just authentication - a logged-in user still cannot reach another user's resource by changing an id; passwords are hashed with bcrypt, scrypt, or argon2, never a fast or raw hash; the session cookie is `httpOnly`, `secure`, and `sameSite`; login and token endpoints are rate-limited. |
| **Secrets & crypto** | no credential, key, or token is hardcoded or committed - it is read from config or secret storage; nothing sensitive reaches a log line or an error returned to the user; crypto uses a vetted library and a current algorithm, never a hand-rolled or deprecated one (MD5/SHA1 for passwords, ECB mode, a static IV/nonce). |
| **Payments / money / PII** | totals reconcile before any write; refunds, fees, and adjustments each have defined behavior; the operation is idempotent on re-run so a retry cannot double-charge; PII never lands in a log; when the inputs do not reconcile the path halts and flags rather than booking the part that matched and skipping the rest. The one question the happy path hides - what happens when the money does not add up - is a done-criterion; `/plan-grill` owns the full failure-behavior pass. |
| **File upload** | type and size are capped before the file is stored; the real type is confirmed by magic bytes, not the extension or the client-sent content-type; the stored file lands outside the web root and is never served executable; the stored name is generated, not taken from the client. |
| **External fetch / SSRF** | a user-influenced URL is allowlisted by scheme and host, and rejected if any *resolved* IP is private, loopback, or link-local (`169.254.169.254` is the cloud-metadata target, the most common SSRF payoff); redirects are not followed blindly. **TOCTOU caveat:** a plain `fetch`/`request` re-resolves DNS *after* your check, so a short-TTL record can rebind between the check and the connect - for a high-risk fetch, resolve once and connect to the pinned IP. Third-party responses are validated as untrusted before use. |
| **Supply chain / postinstall** | a new or bumped dependency is treated as a supply-chain change, not just a version diff: the name is not a typosquat of a popular package (`crossenv` vs `cross-env`), an unfamiliar package's `postinstall`/install script is read before it is allowed to run, and the lockfile is committed so CI resolves the same tree. |
| **LLM / agent feature** | model output never reaches `eval`, SQL, a shell, `innerHTML`, or a file path without validation - it is untrusted input (see the untrusted-output rule in `SKILL.md`); secrets and other tenants' data are kept out of the context window; tool permissions are scoped in code rather than requested in the prompt, and a destructive tool call is confirmed; token count, request rate, and loop/recursion depth are bounded. |
